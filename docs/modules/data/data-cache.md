# 测点数据缓存服务 Data-Cache

## 一、模块介绍

Data-Cache是TBOS动环系统的测点数据缓存服务，负责消费Kafka中的测点数据并缓存到本地内存，为上游服务提供低延迟的测点数据查询能力。

### 1.1 主要职责

| 职责 | 说明                     |
|------|------------------------|
| Kafka测点消费 | 消费主备两路Kafka的测点数据消息     |
| 本地内存缓存 | 将测点数据缓存到本地内存（默认10分钟）   |
| 历史数据预加载 | 服务启动时回溯消费历史数据，确保缓存数据完整 |
| 测点数据查询 | 提供最新值查询、区间查询、变化测点查询等接口 |

## 二、核心能力

### 2.1 主备Kafka双活消费

- **主用Kafka**：`trpc.kafka.tbos.major`
- **备用Kafka**：`trpc.kafka.tbos.backup`（可选配置）
- **消费模式**：批量消费（batch=100，每200ms刷新一次）
- **历史回溯**：启动时自动重置offset到10分钟前（resetSec=600），预加载历史数据
- **Ready机制**：等待所有分区消费到当前时间后才对外提供服务

### 2.2 循环窗口缓存结构

采用分片Map + 循环窗口的缓存结构，高效存储和查询测点数据：

| 组件 | 说明 |
|------|------|
| ShardMap | 分片并发安全Map，默认100000个分片，使用MurmurHash3分片 |
| PointCache | 单个测点的循环窗口缓存，每个窗口存储60秒数据 |
| CompressPoint | 压缩存储的测点，只保存tsOffset(1字节)、quality、value |

**窗口配置**：
- 窗口大小：60秒
- 窗口数量：存储分钟数 + 2个额外窗口（默认63个窗口）
- 数据过期：超过存储时间的数据自动丢弃

### 2.3 测点数据过滤

消费时自动过滤异常数据：

| 过滤规则 | 说明 |
|----------|------|
| 未来数据 | 超过当前时间1分钟的数据被忽略 |
| 历史数据 | 超过1小时前的数据被忽略 |
| 解析失败 | Key或Value JSON解析失败的消息被忽略 |

### 2.4 测点类型支持

| 类型值 | 类型 | 说明 |
|--------|------|------|
| 1 | 采集测点 | 从设备采集的原始测点 |
| 2 | 标准测点 | 经过标准化处理的测点 |
| 3 | 虚拟测点 | 通过计算生成的测点 |
| 4 | 告警测点 | 告警相关的虚拟测点 |

### 2.5 数据质量过滤

查询时支持按数据质量过滤：

| QualityType | 说明 |
|-------------|------|
| QUALITY_TYPE_ALL | 返回所有数据 |
| QUALITY_TYPE_BAD | 只返回质量为负值的数据 |
| QUALITY_TYPE_NORMAL | 只返回质量>=0的数据 |

### 2.6 监控指标

| 指标名 | 说明 |
|--------|------|
| kafka_consume_delay | Kafka消费延迟（毫秒） |
| kafka_consume_msg_cnt | Kafka消费消息数量 |
| kafka_consume_point_cnt | Kafka消费测点数量 |
| local_write_cnt | 本地缓存写入数量 |
| local_write_cost | 本地缓存写入耗时 |
| local_read_cnt | 本地缓存读取数量 |
| local_read_cost | 本地缓存读取耗时 |
| local_read_changed_cnt | 变化测点查询数量 |
| local_read_changed_cost | 变化测点查询耗时 |

### 核心文件说明

| 文件 | 说明 |
|------|------|
| `main.go` | 服务入口，初始化缓存、注册服务、启动Kafka消费并等待Ready |
| `logic/consumer/point_consumer.go` | Kafka消息解析、测点类型识别、数据过滤、指标上报 |
| `logic/kafkasvr/server_transport.go` | 自定义Kafka传输层，支持offset重置和Ready等待机制 |
| `logic/query/point_api.go` | 测点查询逻辑，支持最新值、区间、变化测点三种查询 |
| `repo/cache/local_store.go` | 本地缓存的统一读写入口，包含配置热加载 |
| `repo/cache/point_cache.go` | 单个测点的循环窗口缓存，支持时间范围查询 |
| `repo/cache/shard_map.go` | 泛型分片Map，使用MurmurHash3分片，支持并发安全操作 |

## 四、API接口

### PointService

| 接口 | 说明 |
|------|------|
| `QueryData` | 查询测点数据（最新值或区间查询） |
| `QueryChanged` | 查询指定时间后发生变化的测点 |
| `QueryCachedPoint` | 查询当前缓存的测点列表（支持分页） |

### 查询参数说明

**QueryData**：
- `point_list`：测点名称列表
- `begin`：开始时间戳（0表示查询最新值）
- `end`：结束时间戳
- `quality_type`：数据质量过滤类型

**QueryChanged**：
- `point_list`：测点名称列表
- `begin`：开始时间戳（默认10分钟前）

## 五、常见问题

### 5.1 服务启动慢

**问题描述**：服务启动时需要等待较长时间才能对外提供服务。

**原因分析**：服务启动时会重置Kafka offset到10分钟前，需要消费完历史数据才会Ready。

**解决方案**：
1. 检查日志中的 `kafka finish consume history data` 确认预加载完成时间
2. 如不需要历史数据，可调整配置 `resetSec=0` 跳过历史回溯

### 5.2 查询返回空数据

**问题描述**：查询测点数据返回空结果。

**可能原因**：
1. 查询时间范围超出缓存范围（默认10分钟）
2. 测点名称不存在或拼写错误
3. 服务尚未Ready

**解决方案**：
1. 确认查询时间在缓存范围内（当前时间前10分钟内）
2. 使用 `QueryCachedPoint` 接口确认测点是否已缓存
3. 检查日志确认服务已Ready

### 5.3 查询报错 "read time range over cache duration"

**问题描述**：查询时返回时间范围超出缓存范围的错误。

**原因分析**：请求的begin或end时间超出了配置的缓存时间范围。

**解决方案**：
1. 调整查询时间范围到缓存范围内
2. 如需更长时间范围，修改配置 `store.store_minute` 的值

### 5.4 内存占用过高

**问题描述**：服务内存持续增长或占用过高。

**可能原因**：
1. 缓存时间配置过长
2. 测点数量过多
3. 分片数量配置不合理

**解决方案**：
1. 适当减小 `store.store_minute` 配置值
2. 调整 `store.bulk_cnt` 分片数量
3. 监控 `local_write_cnt` 指标，评估测点规模

### 5.5 Kafka消费延迟过高

**问题描述**：`kafka_consume_delay` 指标持续偏高。

**可能原因**：
1. 上游数据发送延迟
2. 消费处理能力不足
3. 网络问题

**解决方案**：
1. 检查上游数据源的 `pubMs` 时间戳是否正常
2. 增加消费批次大小（batch参数）
3. 检查网络连通性

### 5.6 测点数据被过滤

**问题描述**：日志中出现 `receive data out of range time` 警告。

**原因分析**：测点数据的时间戳不在允许范围内（当前时间前10分钟到后1分钟）。

**解决方案**：
1. 检查上游数据源的时间戳是否正确
2. 确认服务器时间同步正常
3. 检查日志中的sample point信息定位问题测点
